1. PROBLEM STATEMENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Build a modular Inventory Management System (IMS) that digitizes and streamlines
all stock-related operations within a business. The goal is to replace manual
registers, Excel sheets, and scattered tracking methods with a centralized,
real-time, easy-to-use application.

4. AUTHENTICATION MODULE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  4.1 Screens
  ────────────
    • Login Page          – Email + Password fields, "Forgot Password" link
    • Sign Up Page        – Name, Email, Password, Role selection
    • OTP Verification    – 6-digit OTP input, resend timer
    • Password Reset      – Enter new password after OTP verified

  4.2 Flows
  ──────────
    [Sign Up]
      User fills form → POST /api/auth/register → Email OTP sent
      → User verifies OTP → Account activated → Redirect to Dashboard

    [Login]
      User fills credentials → POST /api/auth/login → JWT issued
      → Store JWT in HttpOnly cookie / localStorage → Redirect to Dashboard

    [Forgot Password]
      Enter email → OTP sent → Verify OTP → Set new password → Login

  4.3 Rules
  ──────────
    • JWT access token: 1h expiry; Refresh token: 7 days
    • OTP: 6-digit, valid for 10 minutes, max 3 retries
    • Role-based access control (RBAC): Manager vs Staff permissions
    • Password: min 8 chars, 1 uppercase, 1 number

  4.4 API Endpoints
  ──────────────────
    POST   /api/auth/register
    POST   /api/auth/login
    POST   /api/auth/logout
    POST   /api/auth/send-otp
    POST   /api/auth/verify-otp
    POST   /api/auth/reset-password
    GET    /api/auth/me

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    • Lead time (days)
    • Auto-alert or auto-create receipt draft

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

    Status badges: Ready | Waiting | Done | Cancelled

  8.3 Receipt Detail / Form View (from Mockup)
  ─────────────────────────────────────────────
    Header Fields:
      • Reference No (auto-generated, e.g., WH/IN/00001)

      • Scheduled Date

    Status Bar: [Draft] → [Waiting] → [Ready] → [Done]
    Action Buttons (by status):
      Draft   : [Save] [Discard]
      Waiting : [Validate] [Cancel]
      Ready   : [Validate / Mark as Done] [Return]

      Columns: Product | Description | Qty | Unit of Measure | Qty Done

  8.4 Business Logic
  ───────────────────
    • On Validate (Ready → Done):
        stock_on_hand += qty_received  for each product line
        Move record written to stock ledger
        Receipt status = Done
    • Cannot edit a Done receipt (immutable)
    • Partial receipt: set qty_done < qty_ordered → creates backorder prompt

10. INTERNAL TRANSFERS MODULE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  10.1 Purpose
  ─────────────
    Move stock between locations inside the company.
    Total stock unchanged; location is updated.

  10.2 Examples
  ──────────────
    • Main Warehouse → Production Floor
    • Rack A → Rack B
    • Warehouse 1 → Warehouse 2

  10.3 Transfer Form Fields
  ──────────────────────────
    • Reference No (auto-generated)
    • Source Location
    • Destination Location
    • Scheduled Date
    • Products Table: Product | Qty | UoM

  10.4 Business Logic
  ────────────────────
    On Validate:
      src_location.stock  -= qty
      dest_location.stock += qty
      Move record written to stock ledger

  10.5 API Endpoints
  ───────────────────
    GET    /api/transfers
    POST   /api/transfers
    GET    /api/transfers/:id

11. INVENTORY ADJUSTMENT MODULE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  11.1 Purpose
  ─────────────
    Fix mismatches between system recorded stock and physical count.

  11.2 Steps
  ───────────
    1. Select product & location
    2. Enter counted (real) quantity
    3. System auto-calculates delta (Counted − On Hand)
    4. Confirm → stock updated + adjustment logged

  11.3 Adjustment Record Fields
  ──────────────────────────────
    • Reference No
    • Date
    • Product
    • Location
    • Recorded Qty (system)
    • Counted Qty (physical)
    • Delta (auto-calculated)
    • Reason / Notes

  11.4 Business Logic
  ────────────────────
    delta = counted_qty - recorded_qty
    stock_on_hand += delta  (can be negative for shrinkage)
    Adjustment move written to stock ledger with type = "Adjustment"

  11.5 API Endpoints
  ───────────────────
    GET    /api/adjustments
    POST   /api/adjustments
    GET    /api/adjustments/:id
    POST   /api/adjustments/:id/validate
    POST   /api/adjustments/:id/cancel

12. MOVE HISTORY MODULE (Stock Ledger)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  12.1 Purpose
  ─────────────
    Immutable audit log of every stock movement. Single source of truth.

  12.2 List View (from Mockup)
  ─────────────────────────────
    Columns:
      Date | Reference | Product | From Location | To Location | Qty | Status
    Default: List View
    Filters: Date range | Product | Location | Type | Status

  12.3 Move Record Fields
  ────────────────────────
    • Move ID (unique)
    • Date & Time
    • Reference (links to parent: receipt/delivery/transfer/adjustment)
    • Product
    • From Location
    • To Location
    • Quantity
    • Unit of Measure
    • Move Type   : IN | OUT | INTERNAL | ADJUSTMENT
    • Status      : Draft | Ready | Done
    • Created By (user)

  12.4 Rules
  ───────────
    • Records are immutable once status = Done
    • Every stock change across any module must write a move record
    • Move history is searchable and filterable
    • "A good efficiency move, referred to the Done list" (mockup note)
    • Allow age filtering: display 1 day / 1 week / 1 month

  12.5 API Endpoints

15. STOCK FLOW LOGIC
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Stock balance is always computed from stock_quants table.
  On any validate event, a stock_moves row is written and
  stock_quants is updated atomically (transaction).

  Operation          | stock_quants effect
  ────────────────────────────────────────────────────────
  Receipt Validate   | +qty at destination location
  Delivery Validate  | -qty from source location
  Transfer Validate  | -qty from src, +qty to dest (same total)
  Adjustment Confirm | ± delta at selected location
  ────────────────────────────────────────────────────────

  On Hand    = SUM(quantity)          from stock_quants for product
  Reserved   = SUM(reserved_qty)      from stock_quants for product
  Free to Use = On Hand − Reserved

  Example Walkthrough:
    Step 1: Receive 100 kg Steel            → stock: +100
    Step 2: Transfer: Main Store → Prod Rack → stock unchanged, location updated
    Step 3: Deliver 20 kg Steel             → stock: −20  (balance: 80)
    Step 4: Adjust 3 kg damaged             → stock: −3   (balance: 77)
    All steps logged in stock_moves ledger.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  • Role-based route guards (Manager vs Staff)
  • Input sanitization on all form fields (XSS, SQL injection prevention)
  • HTTPS enforced in production
  • Sensitive data (passwords) hashed with bcrypt (salt rounds ≥ 12)
  • OTP invalidated after single use
  • CSRF protection on state-changing endpoints
  • Rate limiting: 100 req/min per IP on auth endpoints
